Welcome to this lesson on insecure AI API integrations. These days, it is incredibly common for applications to connect to external AI services, whether that is a chatbot, a recommendation engine, or a language model doing something clever in the background. And while these integrations can add real value, they also open up a whole new set of security risks that many developers simply do not think about. Sensitive information can leak out through poorly written API calls, and attackers can actually manipulate the inputs going into an AI system to make it behave in ways it was never supposed to. The consequences of getting this wrong are very real. We are talking about data breaches, falling foul of data protection regulations, and serious damage to your reputation. So throughout this lesson, we are going to look at exactly where these risks come from and what you can do to protect your applications. Let us get started.

Script 2 (Slide 2)
------------------------------------------------------------------------
Let's start by looking at what can actually go wrong when you connect your application to an AI API, because there are quite a few ways this can go badly. First, if your API keys are exposed, an attacker can use your account as if they were you, racking up costs or accessing sensitive data. Speaking of data, every time a user types something into your app and it gets sent off to a third-party AI service, you need to ask yourself: do you actually know what happens to that information on their servers? There's also a clever attack called prompt injection, where a malicious user crafts their input to manipulate what the AI does or says. On top of that, if your integration has overly generous permissions, attackers can pivot into other connected systems. And finally, without proper rate limiting, a hijacked or misconfigured API connection can generate enormous unexpected bills. These are the core risks we'll be unpacking throughout this lesson.

Script 3 (Slide 3)
------------------------------------------------------------------------
What can actually go wrong when AI API integrations aren't properly secured, because the risks are more varied than you might expect. The most obvious issue is exposed API keys. If those fall into the wrong hands, an attacker effectively has the run of your AI service account. Beyond that, the data travelling back and forth in your requests and responses can leak sensitive information if it isn't handled carefully. There are also financial risks to consider. Unauthorised or runaway API calls can rack up enormous costs very quickly. Then there's prompt injection, where an attacker manipulates the input to your AI model to make it behave in ways you never intended. If you're not logging and monitoring your API activity, you'll have no record of what happened when something goes wrong. And finally, if sensitive user data is crossing unsecured API boundaries, you could find yourself in serious trouble from a compliance and regulatory standpoint.

Script 5 (Slide 4)
------------------------------------------------------------------------
One of the most common mistakes developers make when working with AI APIs is hardcoding their API keys directly into their source code. It looks harmless enough, but the moment that code is pushed to a repository, even a private one, that secret is exposed. Repositories get shared, forked, made public by accident, or simply accessed by someone who should not have that level of visibility. And here is the thing: automated bots are constantly scanning platforms like GitHub, looking specifically for exposed credentials. We are talking minutes, sometimes seconds, before a leaked key is picked up and abused. If you suspect a key has been compromised, rotate it immediately through your API provider's dashboard and revoke the old one. Going forward, the right approach is to store secrets in environment variables, keeping them completely out of your codebase. That one habit alone will protect you from a surprisingly large number of security incidents.

Script 6 (Slide 5)
------------------------------------------------------------------------
So now that we understand the risks, let's talk about what you can actually do to protect yourself. First, make sure every connection to an AI API goes over HTTPS, never plain HTTP, because that encrypts the data in transit and stops anyone snooping on your requests. Next, think about your API keys like passwords. You should be rotating them on a regular basis, and if a key is ever exposed, you can revoke it quickly without too much damage. It is also worth restricting what each key is actually allowed to do. If a key only needs read access, do not give it write permissions as well. Once data comes back from the API, do not just trust it blindly. Validate those responses to make sure they look as expected before your application acts on them. And finally, keep logs of your API calls and monitor them for anything unusual. These habits together will significantly reduce your exposure to the risks we have been discussing.

Script 8 (Slide 6)
------------------------------------------------------------------------
Here are some core principles to carry forward. First, always treat data moving in and out of an AI service as potentially dangerous, which means validating and sanitising it properly. Keep your API keys out of client-side code and well away from public repositories like GitHub. And here is something people often overlook: just because an AI has generated a response does not mean you should trust it blindly. Treat that output the same way you would treat any unverified input. When you set up access and permissions, give the integration only what it absolutely needs, nothing more. Make sure you are logging your API calls so you can spot anything unusual before it becomes a real problem. And finally, do not rush to production. Start small, and check your integration against a proper security checklist first. These principles are straightforward, but following them consistently is what separates a secure integration from a vulnerable one.
